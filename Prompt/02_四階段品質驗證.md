# 02 四階段品質驗證

- **論文章節**：§參-五.1 三元組品質驗證（四階段自動化品質驗證管線）
- **用途**：以混合式過濾漏斗架構（Hybrid Funnel Architecture）對萃取出之 RawTriples 逐筆把關後匯入 Neo4j。借鏡 Regino & dos Reis (2025) 之 LLM-based RDF 驗證四任務，重組為硬性檢查在前、語意檢查在中、寫入在後之分層管線。
- **模型**：三段 LLM 呼叫皆使用 `meta/llama-3.3-70b-instruct`（NVIDIA）
- **參數**：`temperature=0.0`、`response_format={"type":"json_object"}`
- **來源檔案**：
  - `論文\CybersecurityLearningPlatform\backend\ETL_module\03_validate_and_import_fixed.py`
    - 共用系統提示詞：`call_llama()` 第 194 行
    - `prompt_1`（Step 1 類別與屬性對齊）：第 390 行起
    - `prompt_2`（Step 2 實體對齊與 URI 標準化）：第 458 行起
    - `prompt_3`（Step 3 語意一致性）：第 524 行起
  - 對照彙整：`論文\CybersecurityLearningPlatform\LLM_PROMPTS.md` §3

## 驗證流程概覽

```
Phase 1（靜態）   : Schema Edge 合法性檢查（程式碼，無 LLM）
Phase 2（無鎖並行）: Step 1 類別/屬性對齊  →  Step 3 語意一致性檢查（兩步皆為 LLM）
Phase 3（加鎖）   : Step 2 實體 URI 標準化與去重（LLM + 向量語意相似度）→ 寫入 Neo4j
```

> 命名說明：程式以 `validation_history['step_1']`、`['step_3']`、`['step_2']` 紀錄結果。Step 1 與 Step 3 為無狀態並行（Phase 2）；Step 2 需查詢 Neo4j 現有節點，故置於加鎖區段（Phase 3）。三段共用同一系統提示詞。

## Step 1 — 類別與屬性對齊（Class and Property Alignment）

- **驗證歷史鍵**：`validation_history['step_1']`；觸發條件：Phase 1 靜態 Schema 檢查通過後立即執行。
- **輸出格式**：JSON，含 `step_1_list_check`、`step_2_alignment_check`、`response`（`correct` / `violation`）、`reason`。

```text
你是一位嚴謹的資安本體論專家。你的任務是執行「類別與屬性檢查 (Class and Property Alignment)」，確認以下三元組是否完全符合預先定義的類別與屬性清單 (Lc,p)。

                ### 待檢查三元組：
                - 主體 (Subject): {subject_name} (類別: {subject_type})
                - 關係 (Relation): {relation}
                - 受體 (Object): {object_name} (類別: {object_type})

                ### 合法清單 Lc,p：
                {Lc_p}

                ### 判斷準則（依序執行）：
                1. 清單合法性：主體的類別與受體的類別是否都存在於 Lc,p 的 "Classes" 中？關係是否存在於 "Properties" 中？
                2. 語意對齊：每個實體的「名稱」與其被賦予的「類別」是否合理對齊？
                3. 同義詞寬容原則：請接受資安領域常見的同義詞、縮寫與翻譯。

                ### 輸出格式（JSON，思維鏈）：
                {
                "step_1_list_check": "說明主體類別、受體類別與關係是否皆在 Lc,p 中。",
                "step_2_alignment_check": "判斷各實體名稱是否為其類別的合理實例（含同義詞寬容）。",
                "response": "correct 或 violation",
                "reason": "綜合上述步驟的一句話結論。"
                }
```

> `{Lc_p}` 執行時以 `json.dumps(Lc_p, ensure_ascii=False, indent=2)` 展開，內容為 15 個合法 Classes 與 16 個合法 Properties 的 JSON 字典。

## Step 2 — 實體對齊與 URI 標準化（URI Standardization）

- **驗證歷史鍵**：`validation_history['step_2']`；觸發條件：Phase 2 兩步皆通過後，於加鎖區段（`step2_and_write_lock`）內執行。
- **前置步驟**：先呼叫 `queryForDuplicateResources()`，以語意向量（EmbeddingGemma）從 Neo4j 取得近似實體清單 `{Ldr}`（cosine > 0.3，top-k=10），再注入提示詞。
- **輸出格式**：JSON，含 `granularity_analysis`、`response`（`duplicate` / `correct`）、`standard_subject`、`standard_object`、`reason`。
- **後置動作**：若 `response == "duplicate"`，以 `standard_subject`／`standard_object` 取代原始名稱後寫入 Neo4j。

```text
你是一位嚴謹的資安本體論專家。你的任務是執行「實體對齊與資源重複檢查 (URI Standardization)」。

                    ### 待檢查三元組：
                    - 主體 (Subject): {subject_name} (類別: {subject_type})
                    - 受體 (Object): {object_name} (類別: {object_type})

                    ### 現有近似資源清單 (Ldr)：
                    {Ldr}

                    ### 判斷準則（🚨 極度重要）：
                    1. 【階層嚴格保留，禁止合併】：若兩者存在廣義/狹義的從屬、種類或包含關係（例如：「惡意程式」與「木馬程式」、「攻擊」與「Web攻擊」），這代表不同的知識分類階層(Taxonomy)，絕對禁止將其視為重複，直接判定 correct。
                    2. 【領域同義詞寬容，鼓勵合併】：對於資安與 IT 領域中高度重疊的實體（例如：「iOS 手機」與「行動裝置」、「漏洞」與「弱點」、「Smartphone」與「智慧型手機」），請視為同義詞並判定為 duplicate，進行標準化。
                    3. 【前提確認】：若 Ldr 為空，或實體名稱與 Ldr 中某資源完全相同，代表已是標準名稱，直接判定 correct。

                    ### 輸出格式（JSON，思維鏈）：
                    {
                    "granularity_analysis": "分析兩者是純粹的同義詞(適用準則2)？還是廣義/狹義的從屬層級關係(適用準則1)？",
                    "response": "duplicate 或 correct",
                    "standard_subject": "若主體重複，填入 Ldr 的標準名稱；否則留空。",
                    "standard_object": "若受體重複，填入 Ldr 的標準名稱；否則留空。",
                    "reason": "一句話簡潔總結。"
                    }
```

> `{Ldr}` 執行時以 `json.dumps(Ldr, ensure_ascii=False, indent=2)` 展開，各項含 `name`、`type`、`similarity_score`。
> 對應論文：第二階段以 embedding 餘弦相似度進行 top-k 候選篩選並搭配 0.80 相似度閾值，於送入 LLM 前先縮減候選重複資源集合 Ldr 之規模。

## Step 3 — 語意一致性（Semantic Consistency）

- **驗證歷史鍵**：`validation_history['step_3']`；觸發條件：Step 1 通過（`response == "correct"`）後執行。
- **輸出格式**：JSON，含 `step_1_rule_matching`、`response`（`correct` / `violation`）、`reason`。

```text
你是一位嚴謹的資安本體論專家。你的任務是執行「語義一致性檢查 (Semantic Consistency)」，判斷以下三元組是否違反語義限制清單 (Lsr) 中的任何規則。

                ### 待檢查三元組：
                - 主體: {subject_name} (類別: {subject_type})
                - 關係: {relation}
                - 受體: {object_name} (類別: {object_type})

                ### 語義限制清單 (Lsr)：
                {Lsr}

                ### 驗證邏輯：
                1. 遵守即合法：若三元組符合規則所允許的條件，代表合法，絕對不可判定為違規。
                2. 無罪推定：若未發現「明確且直接」的違反項目，response 必須為 correct。

                ### 輸出格式（JSON，思維鏈）：
                {
                "step_1_rule_matching": "針對每條相關規則，說明三元組是否觸發違規。",
                "response": "correct 或 violation",
                "reason": "最終判定理由（不超過兩句話）。"
                }
```

> `{Lsr}` 執行時以 `json.dumps(Lsr, ensure_ascii=False, indent=2)` 展開，內容為以下 8 條靜態語意限制規則：
> 1. 同一實體不能同時具有攻擊者(attacker)與安全團隊(securityTeam)的身分。
> 2. 主體為 policy, technique 或 tool 時才可執行 mitigates 動作。
> 3. 主體類別若為 vulnerability (漏洞)，原則上不可發起主動行為，【特例允許】can_expose。
> 4. principle (資安原則) 僅能作為 violates 的受體，不可作為主體。
> 5. 關係為 uses 時，主體 (Subject) 的類別不可為 data。
> 6. 關係為 deployed_in 時，受體必須是 system。
> 7. 關係為 generates 時，受體必須是 data。
> 8. 關係為 controls 時，主體必須是具有主動執行能力的角色或工具（attacker, securityTeam, tool）。

## 變數說明

- `{subject_name}`／`{subject_type}`／`{relation}`／`{object_name}`／`{object_type}`：待驗證三元組之欄位。
- `{Lc_p}`：合法類別與屬性清單（15 Classes＋16 Properties）。
- `{Ldr}`：由向量相似度從 Neo4j 查得之近似節點候選清單。
- `{Lsr}`：8 條語意限制規則清單。

## 設計對應（論文敘述）

- 第一階段先以程式進行結構與合法邊界關聯檢核（Phase 1），再由 Llama-3.3-70B 補語意對齊（Step 1）。
- 第二階段對應原研究 URI Standardization，以 embedding top-k＋0.80 閾值控制 Ldr 規模後再交 LLM 判定同義並標準化。
- 第三階段依 Lsr 規則進行語意一致性審查。
- 語法／資料結構驗證合併入第一階段（Phase 1 靜態檢查）。
